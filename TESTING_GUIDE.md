# Testing Guide for the NCC C Compiler

## Running Tests

### Full Test Suite
To run all tests, execute the following command from the project root:
```bash
tests/run_tests.sh --no-preprocess build/src/ncc
```
For the sake of time, unless specifically trying to check something relating to preprocssing, use the --no-preprocess 
flag.

### Single Test
To run a specific test file (e.g., `tests/test_bitwise.c`):
```bash
tests/run_tests.sh --no-preprocess build/src/ncc tests/test_bitwise.c
```

## Test Output Files

For each test file `tests/XXX.c`, the test script generates several files in the `tests/output/` directory:

| File | Description |
|------|-------------|
| `tests/output/XXX.ll` | LLVM IR generated by ncc (from `-S --emit-llvm`) |
| `tests/output/XXX.err` | Combined stderr output from ncc and clang compilation |
| `tests/output/XXX.tmp.o` | Temporary object file generated by ncc (`-c` flag) |
| `tests/output/XXX` | Linked executable (if linking succeeds) |
| `tests/output/XXX_link` | Alternative name for the linked executable (used during linking step) |

### Example
For `tests/test_bitwise.c`, you would see:
- `tests/output/test_bitwise.ll`
- `tests/output/test_bitwise.err`
- `tests/output/test_bitwise.tmp.o`
- `tests/output/test_bitwise` (the executable)

## Understanding Test Results

The test script considers a test **passed** if:
1. ncc successfully compiles the C file to LLVM IR and then to an object file
2. clang successfully links the object file into an executable
3. The executable runs and exits with code 0

A test is **failed** if any of the above steps fail.

## Preprocessing Features

As of the latest update, the NCC compiler now includes preprocessing support by default, using clang's preprocessor. This means:

- By default, all input files are preprocessed before compilation
- The `-E` flag outputs preprocessed code to stdout and stops (clang-compatible)
- The `--no-preprocess` flag disables preprocessing, preserving the original behavior
- `-I` and `-D` flags are passed through to the clang preprocessor

### Preprocessing-related Command Options

| Option | Description |
|--------|-------------|
| `-E` | Preprocess only; output result to stdout |
| `--no-preprocess` | Disable preprocessing (use original behavior) |
| `-I <dir>` | Add directory to include search path (passed to preprocessor) |
| `-D <macro>` | Define preprocessor macro (passed to preprocessor) |

### Example Usage

```bash
# Default behavior: preprocess then compile
build/src/ncc -o program program.c

# Preprocess only (output to stdout)
build/src/ncc -E program.c

# Compile without preprocessing (original behavior)
build/src/ncc --no-preprocess -o program program.c

# With include paths and defines
build/src/ncc -I./includes -DDEBUG -o program program.c
```

## Useful Notes from Development

### Compiler Architecture
- The compiler uses a GDL-based parser that generates an AST
- LLVM IR is generated via the LLVM C API
- Symbol table management is scope-based (hierarchical)
- Function pointers are fully supported, including:
  - Declaration: `int (*fp)(int, int);`
  - Assignment: `fp = &func;`
  - Indirect calls: `(*fp)(1, 2);`
  - Arrays of function pointers: `int (*ops[2])(int, int);`

### Key Implementation Details
- **Type Mapping**: The `map_type()` function in `llvm_ir_generator.c` handles conversion from C types to LLVM types
  - `_Bool` and `bool` map to `LLVMInt1TypeInContext` (1-bit integer)
  - Standard types: `char`→i8, `short`→i16, `int`→i32, `long`→i64
- **Boolean Literals**: The identifiers `true` and `false` are handled specially in `process_identifier()`
  - `true` → `LLVMConstInt(i1, 1, false)`
  - `false` → `LLVMConstInt(i1, 0, false)`
- **Type Promotion**: In expressions involving different integer types, the compiler promotes narrower types to wider ones using zero extension (`ZExt`) for unsigned values
- **Return Statement Handling**: When returning a boolean value (`i1`) from a function declared to return `int` (`i32`), the compiler automatically zero-extends the value to match the return type

### Debugging Tips
- To see the LLVM IR for a test: `build/src/ncc -S --emit-llvm -o - tests/test_file.c`
- To see just the assembly: `build/src/ncc -S -o - tests/test_file.c`
- The test script preserves intermediate files in `tests/output/` for inspection
- Error messages from ncc and clang are captured in `*.err` files

### Common Issues Encountered
1. **Type Mismatch in Return Statements**: Fixed by ensuring boolean values (`i1`) are properly zero-extended to match the function's return type (usually `i32`)
2. **Bitwise Operations on Booleans**: Required promoting boolean operands to match the width of other operands in the expression
3. **Function Pointer Array Allocation**: Fixed by correctly extracting array sizes from within `FunctionPointerDeclarator` AST nodes
4. **Scope Management**: Ensured symbol lookups respect lexical scope (inner scopes can shadow outer ones)

## Troubleshooting

If a test fails:
1. Check `tests/output/XXX.err` for compilation errors
2. Examine `tests/output/XXX.ll` to see the generated LLVM IR
3. Verify the test C file is valid and expects exit code 0
4. For link errors, check if the object file (`XXX.tmp.o`) was generated correctly
5. For runtime errors, run the executable directly: `tests/output/XXX`

## Notes relating to AST actions
- Rules in the grammar definition may have AST semantic actions assigned to them by adding @<ACTION_NMAE_GOES_HERE> at 
the end of a rule (before the semicolon). Note that the action is assigned to the rule (the LHS of the rule), not to portions of the rule.
That means this sort of thing isn't supported...
```
FloatLiteralBase = ((DigitSeq? '.' DigitSeq Exponent?) @AST_ACTION_FOR_THIS_PORTION_IS_ILLEGAL
                 | (DigitSeq '.' Exponent?)
                 | (DigitSeq Exponent));
```
To assign an action just to that portion of a rule, that portion must be split out into its own rule. e.g.
```
FloatLiteralBasePortion = ((DigitSeq? '.' DigitSeq Exponent?) @AST_ACTION_IS_LEGAL;
FloatLiteralBase = FloatLiteralBasePortion
                 | (DigitSeq '.' Exponent?)
                 | (DigitSeq Exponent));
```
- Once the action is assigned a callback handler for that action must be registered in c_grammar_ast_actions.c.
Refer to existing handler registrations in that file for examples.
- The handler itself receives a list of AST actions that have been pushed onto a stack by other handler callbacks that 
have been called during the AST walk. 
- Note that these are _exit_ handlers, so the children nodes will be added by 
children of the current callback action.
- Note that some CPT nodes may _not_ have actions assigned to them, so the input they match on will be consumed, but not
presented in the AST callback.
- The CPT node that is assocaiated with the rule the semanic action is assigned to _is_ presented to the callback as a
parameter.
- Each rule may only have one action assigned to it.
- Note that assigning an action to a rule that didn't previously have one _may_ break handling of the AST nodes in
llvm_ir_generator.c because there are places where the code expects a child node to be present in a specific position.
- Note that a callback handler must either consume the child nodes it is presented with, or free them once it has 
finished with them. By 'consume', I mean either push it back onto the AST stack, or associate it with a new node that 
is then pushed back onto the stack. e.g. a child node might be assigned to a new node's 'lhs' field.
